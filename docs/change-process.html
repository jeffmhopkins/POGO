POGO · CHANGE PROCESS

← docs  ·  how the project is modified

Every change is plugin-first and lives on its own change/<slug> branch off dev, merging back by PR. Never commit directly to dev or main. The VCV Rack plugin is the ground truth; the hardware spec and KiCad schematics follow the locked plugin and never lead it.

Source of truth: this page summarizes CLAUDE.md → “Git Workflow & Change Process”. If the two ever disagree, CLAUDE.md wins.

Pick a lane first

LaneWhenPath
A — Behavioralchanges audible DSP or panel-control behaviorfull Steps 0–8, gates G1–G6
B — Hardware-onlyplugin already locked; only spec §2–4 / nets / components / schematicenter at Step 5 (G4–G6 + --check)
C — Trivialdocs / comments / tests / typos; no DSP math, panel geometry, components.yaml, or net connectivityminimal change file + a PR that passes CI

If unsure whether behavior changes, use Lane A. A “refactor” that alters audio output is a Lane A change. If hardware work uncovers a plugin bug, that spawns a separate Lane A change — the plugin always leads.

Conventions

  • Branch: change/<slug>, one change per branch/PR.
  • Change file: every change gets a persistent changes/NNNN-<slug>.md (from changes/_TEMPLATE.md), committed as the first commit on the branch, before the PR, and never deleted (NNNN = highest on dev + 1).
  • PR title: NNNN-slug: summary.

Steps & gates (Lane A)

  1. Open — branch + change file.
  2. Intent — you state audio/UX intent; assistant restates. G1.
  3. Plugin + Panel — DSP + panel-data.yaml, DRC clean, push. G2 = panel layout.
  4. Build + Verify — CI builds the .vcvplugin; you verify in VCV Rack. G3.
  5. Lock — record the locked plugin/panel blob hashes; they freeze for the change.
  6. Spec — block §1 intent + functional spec to the locked behavior. G4.
  7. Topology + components — topology prose; new parts approved into components.yaml + registry + footprint. G5 / G6.
  8. Schematic + artifacts — netlists → .kicad_sch, BOM, panel; --check clean.
  9. Close — STATUS.md + change file; PR → dev; squash-merge on green CI.
G1 intent G2 panel G3 behavior G4 spec G5 topology G6 component+footprint

Gates are hard stops — work waits for explicit approval. On rejection: log the reason, revert to the last approved state, and re-enter at the failed step.
